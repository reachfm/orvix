## 1. Current architecture health

Go 1.25 backend (Fiber v3, GORM, pgx, go-redis, zap), React frontend, modular `internal/` packages (~50 packages). Builds clean, vets clean, all tests pass. The module boundary between "enterprise metadata" (Postgres-capable) and "CoreMail operational data" (SQLite-only) is real and consistently applied — this is a legitimate, intentional architecture, not confusion. No evidence of the redesign-without-reason the brief warns against having occurred.

**Health: GOOD**, with one structural caveat carried through nearly every subsystem below: **CoreMail's SQLite store is hard-pinned to a single DB connection** (`internal/coremail/storage/mailstore.go:45-46`, `SetMaxOpenConns(1)`) plus an application-level mutex, because the pure-Go `modernc.org/sqlite` driver doesn't reliably honor `busy_timeout`. Every IMAP/POP3/webmail read and every delivery write for the **entire server** serializes through one connection. It has never been load-tested — `internal/storage/loadtest/loadtest_test.go` states in its own source: *"THIS PACKAGE DOES NOT YET CONNECT TO A REAL DATABASE... simulation-only."* This is the single largest scalability question mark in the codebase.

## 2. Security health

Mixed. Foundations are good; enforcement is inconsistent.

- **Password hashing — SOLID.** bcrypt for admin/user accounts, Argon2id for mailbox passwords. No MD5/plain-SHA256 password storage anywhere.
- **SQL injection — SOLID.** All dynamic SQL uses parameterized placeholders; no raw string concatenation of user input found.
- **MFA — PARTIAL.** Real RFC-6238 TOTP, real secret generation and QR/otpauth URL — but the secret is stored **reversibly** (base64, `admin_mfa.go:95-96,160-162`) despite a code comment claiming it's hashed. DB compromise = attacker can generate valid codes indefinitely.
- **CSRF — WEAK.** Real double-submit implementation exists but is opt-in per route, not global (`internal/api/router.go`). Several state-changing admin endpoints bypass it entirely: migration start, domain provisioning, calendar/contacts/tasks CRUD, update-apply, compliance policy writes. Compounding this, auth cookies are set `SameSite: "None"`, which *helps* cross-site requests reach these unprotected endpoints rather than blocking them.
- **Session handling — PARTIAL.** RS256 JWT with algorithm-confusion guard, sane TTLs (15 min access / 30 day refresh), real session-table revocation — but stateless access tokens aren't checked against any blocklist on logout, so a stolen token survives up to 15 minutes post-logout. Bounded risk, not critical.
- **Audit logging — PARTIAL.** Correct shape (actor/role/action/target/ip/timestamp) but it's a plain mutable table with a `PurgeOlderThan` deletion path — no hash-chaining or append-only enforcement. Anyone with DB write access can edit or erase entries undetected.

### 🔴 CRITICAL — Cross-tenant IDOR / account takeover
Every mailbox/domain admin handler in `internal/api/handlers/handlers.go` scopes reads/writes by numeric ID only, never re-checking `tenant_id` (even though the column exists and is set correctly at creation):
- `UpdateMailboxPassword` — `handlers.go:1926,1940`
- `DeleteMailbox` — `handlers.go:2280,2293`
- `GetMailbox` — `handlers.go:1486`
- `GetDomain` — `handlers.go:1210-1216`

**Any authenticated admin in Tenant A can reset the password of, or delete, a mailbox belonging to Tenant B** by guessing/incrementing an ID. For a multi-tenant product, this breaks the core isolation guarantee and is a live, exploitable vulnerability today — not a hypothetical.

## 3. Remaining PostgreSQL limitations

Feature-complete for metadata as claimed, and evidence-backed (schema/migration/DML/backup/restore/CI all genuinely pass). But: **the production backup service never dumps the Postgres metadata DB.** `internal/backup/service.go` (1723 lines) only backs up the SQLite CoreMail store (`VACUUM INTO` + tarballs) — there is no `pg_dump` call anywhere in the actual backup path. A `pg_dump`/`pg_restore` integration test exists (`cmd/orvix/main_test.go:556`) but it's gated behind `ORVIX_RUN_POSTGRES_BACKUP_TEST=1` and isn't wired into the real backup service. **Users, tenants, licenses, and auth data currently have no automated backup on a Postgres deployment.**

Per your instruction, this is a real production blocker discovered during the audit — flagging it, not fixing it yet.

## 4. CoreMail architecture status

Filesystem-backed message store (`.eml` files under `{tenant}/{domain}/{mailbox}/{msgID}.eml`, metadata + path + SHA256 in DB) — a sound design, not SQLite BLOBs. But: writes are **not atomic** (direct `os.WriteFile`, no temp-file+rename, no fsync — a crash mid-write can leave a truncated message), and **oversized/excess attachments are silently dropped** (`mailstore.go:241-249`, a `continue` with no error surfaced and no bounce generated). Functionally works end-to-end; these are real data-integrity defects, not cosmetic gaps.

## 5. SMTP readiness — SOLID (one gap)

Real inbound (EHLO/MAIL/RCPT/DATA, STARTTLS, size/recipient limits) and real outbound (MX lookup, EHLO→STARTTLS→DATA client, submission on 587 with mandatory STARTTLS). Compiles clean, and its test suite runs genuine loopback TCP conversations, not mocks.

**Gap:** SMTPS (port 465, implicit TLS) is explicitly unimplemented — `runtime/module.go:379-385` logs a warning and never starts the listener, despite the config flag existing and being exposed to admins (operator-trust gap).

DKIM/SPF are SOLID (real RSA signing/verification, real recursive SPF evaluation against live DNS). One landmine: `dkim/signer.go`'s `Verify()` method is dead code that unconditionally returns `true` without checking the signature — unused in production today, but dangerous if anyone wires it in later; should be deleted or fixed.

DMARC is **PARTIAL** — real DNS lookup, alignment, and RFC 7489 pass/fail logic, but `p=reject` is never enforced directly; it's diluted into a weighted spam score (6.0 of an 8.0 reject threshold), so a domain publishing a strict reject policy can still have spoofed mail accepted. Antispam is **PARTIAL** — 10 real heuristic rules, but zero content/body inspection; the admin API itself reports `"antispam_engine": "rspamd_not_wired"`. Antivirus (ClamAV) is SOLID — real clamd wire protocol, not a no-op.

## 6. IMAP readiness — PARTIAL, not RFC 3501-compliant

Implemented: `CAPABILITY, NOOP, LOGOUT, LOGIN, LIST, SELECT, STATUS, FETCH, STORE, COPY, EXPUNGE, UID`. **Missing entirely: APPEND, SEARCH, IDLE, EXAMINE, CREATE/DELETE/RENAME/SUBSCRIBE mailbox management, STARTTLS-on-143.** No literal TODOs — these are clean omissions, not flagged debt.

This is the most consequential protocol gap in the whole audit: without APPEND, clients can't save to Sent/Drafts over IMAP; without SEARCH, every client falls back to fetch-and-filter; without IDLE, there's no push. **No mainstream enterprise mail client (Outlook, Thunderbird, Apple Mail) will behave acceptably against this today.** Test coverage is real (genuine wire-protocol round-trips over TCP) but only covers the narrow command set that exists.

## 7. POP3 readiness — SOLID

Full baseline: `USER, PASS, STAT, LIST, RETR, DELE, UIDL, TOP, NOOP, RSET, QUIT`, TLS on 995. Missing APOP and STLS (opportunistic TLS upgrade), but this is minor given TLS-wrapped 995 covers the real-world need. The most complete of the three mail-access protocols relative to its (small) spec.

## 8. MailStore readiness — PARTIAL

See §4. Functionally solid for basic send/receive/read; atomicity and silent-attachment-drop are the open defects. Best-tested subsystem in the audit (2,436 lines of storage tests including real pagination/scale benchmarks to 10k rows).

## 9. Queue readiness — PARTIAL

The most solid engineering in CoreMail: real exponential backoff (1m→5m→15m→1h→4h→24h), dead-letter transition, lease-based worker claiming with expiry recovery, admin retry/requeue. **Gap:** "bounce" is status-only — `Bounce()` just flips a DB column. Zero RFC 3464 (`multipart/report`) generation anywhere; no actual DSN email is ever sent to the original sender when a message permanently fails.

## 10. Search readiness — NOT STARTED

Confirmed: zero full-text search library anywhere (go.mod, go.sum, and full-repo grep for bleve/meilisearch/elasticsearch/tantivy/fts5 all come back empty). "Search" today is unindexed SQL `LIKE '%term%'` against header columns only. Worse: there's a `SearchBody` flag threaded from the webmail UI through to the storage layer's `MessageFilter`, but the storage layer **never actually checks message bodies** — confirmed dead by the storage layer's own test comment. This is not a partial feature; it's a UI-visible option that silently does nothing. Roadmap Phase 8 has no code behind it. This is a hard, visible competitive gap against Exchange (Content Index) and Zimbra (Lucene).

## 11. Enterprise installer readiness — PARTIAL

Genuinely strong for a curl-pipe installer: dedicated unprivileged `orvix` system user, sha256/provenance verification, bcrypt-enforced admin bootstrap, idempotent re-run via surgical config migration, hardened systemd unit (`NoNewPrivileges`, `ProtectSystem=full`, minimal capability set), tightly scoped sudoers. **Gap:** firewall (ufw) and TLS/ACME setup are **not** run by `install.sh` — they require a separate manual script (`setup-https.sh`). The advertised `ORVIX_SETUP_HTTPS`/`ORVIX_HARDEN_FIREWALL` env vars are exported but never actually read by `install.sh` — dead configuration, a real doc-vs-code mismatch. Until the manual step runs, HTTP listeners are loopback-only, which is safe-by-default but not "one command, done."

## 12. Upgrade readiness — SOLID

Real pre-flight backup (binary, config, SQLite DB, JWT/VAPID/DKIM keys), checksum verification, health-check gating, and automatic rollback on any failure. Self-documented caveat: it can't snapshot a running SQLite DB without a coordinated `VACUUM INTO`, so it depends on the runtime backup feature for a true point-in-time mail-store snapshot — consistent with, and compounding, the §3 backup gap.

## 13. Multi-tenant readiness — CRITICAL GAP

See §2. `tenant_id` exists and is set correctly at creation but is not enforced on read/update/delete for mailboxes and domains. This must be fixed before Orvix can be sold as a genuine multi-tenant platform — as-is, one tenant can take over another tenant's mailbox.

## 14. HA readiness — NOT STARTED (honestly self-disclosed)

No clustering, replication, or failover code anywhere (confirmed by full-repo grep; other hits were false positives in unrelated draft-mail fields). The product's own admin API is honest about this: `enterprise_admin_v3.go:2115` returns *"Orvix Enterprise currently runs as a single-node deployment. Clustering + proxy replication is not implemented in this build."* Good that it's disclosed rather than faked; still a hard requirement for "Enterprise HA roadmap" and needs its own dedicated architecture phase (same treatment as the CoreMail-to-Postgres question).

## Phased execution plan — current state → Enterprise v1

This revises the existing Phase 1–17 roadmap by inserting a **Phase 0 (Security & Process Closure)** ahead of everything else, per your instruction to treat real production blockers as an exception to "no more Postgres work." Phases 1 (DB) and 2 (CoreMail) are re-scoped to reflect what's actually done vs. claimed. No phase is skipped; each carries acceptance criteria and a release gate.

### Phase 0 — Security & Process Closure (NEW, blocks everything else)
**Scope:** Blockers #1, #2, #3, #7, #14, #16 from §15.
**Milestones:** tenant-scoped query layer for all mailbox/domain CRUD; Postgres metadata backup wired into `internal/backup/service.go`; global CSRF enforcement; MFA secret encrypted at rest; branch cleaned and merged to `main` via proper PR.
**Acceptance criteria:** a cross-tenant access attempt returns 403/404 in an automated regression test for every mailbox/domain handler; a restore-from-backup drill recovers both the Postgres metadata DB and the SQLite mail store; every state-changing admin route has CSRF coverage verified by an integration test that asserts rejection without a valid token.
**Testing:** new IDOR regression suite, backup/restore drill on a throwaway VPS, CSRF integration tests, full existing suite must still pass (`go test -p 1 -count=1 ./... -timeout=1200s`).
**Release gate:** no PR after this phase may merge without the tenant-scoping helper being used; security review required on every handler touching mailbox/domain/user IDs.

### Phase 1 — Enterprise Database (re-scoped: bug-fix-only, as instructed)
Already feature-complete for metadata per real evidence in §0/§3. Remaining work is limited to the backup gap (moved to Phase 0) and any bugs found in production. No new Postgres features until explicitly approved.

### Phase 2 — CoreMail Engine (in progress, re-scoped)
**Scope:** Blockers #4 (IMAP APPEND/SEARCH/IDLE), #10 (silent attachment drop, non-atomic writes).
**Milestones:** IMAP passes a real client-compatibility smoke test (Thunderbird: save-to-Sent works, server search works, new mail appears without manual refresh); message writes are crash-safe (kill-mid-write test recovers cleanly); oversized attachments are rejected with a clear SMTP error, never silently dropped.
**Acceptance criteria:** automated IMAP protocol test suite covering APPEND/SEARCH/IDLE; a fault-injection test that kills the process mid-write and verifies no truncated `.eml` on restart.
**Testing:** protocol-level wire tests (already the project's established pattern), fault injection, existing regression suite.
**Release gate:** at least one real mail client (Thunderbird or Outlook) demonstrated working end-to-end against a staging Orvix instance, screenshots/log attached to the PR.

## Immediate recommendation

Do not resume PostgreSQL feature work. Start Phase 0 now: it contains the one item that already qualifies as your stated exception ("real production blocker") — the cross-tenant IDOR — plus the equally critical missing Postgres-metadata backup. Both are cheap to scope (days, not weeks) and both are load-bearing for calling this an "enterprise" product at all. Recommend opening two separate branches/PRs for #1 and #2 so they can be reviewed and merged independently and quickly, per the required feature-branch → tests → PR → review → merge workflow — and getting `db/postgres-final-closure` itself cleaned up and merged to `main` in parallel, since no new work should stack on an unmerged, dirty branch.
